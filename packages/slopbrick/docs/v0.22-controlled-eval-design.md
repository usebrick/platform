# v0.22 controlled-eval design: `slop_suggest_with_structure` agent conformance

**Author:** slopbrick project, 2026-07-01
**Status:** v0.22 design (the controlled-eval harness is the sharpest gap vs. VibeDrift's published eval per the strategic review)
**Audience:** ML eval engineers, anyone measuring whether AI coding agents actually *use* the structure information we ship

---

## TL;DR

slopbrick ships an MCP tool, `slop_suggest_with_structure`, that gives an AI agent a fast-path read of `.slopbrick/structure.md` (or a re-scan fallback). The tool's *correctness* is tested by the existing v8.5 calibration (rule precision/recall on a 546k-file corpus). What is **not** tested is whether an AI agent that *receives* the tool's output actually produces code that conforms to the structure the tool describes.

This is the gap. Rules can be calibrated to 30x lift and still produce zero behavioral change in agents that ignore the tool. The v0.22 controlled-eval is a harness that measures the gap, with vs. without the tool, on a fixed set of coding tasks in fixed codebases, with a fixed conformance metric.

The harness exists in `tests/research/controlled-eval.test.ts` as a stub (test cases + conformance check + agent stub). It does not yet call a real LLM. The stub returns a fixed output so the test framework can be validated end-to-end before the LLM wiring is added.

---

## The gap

Three layers of evidence, each well-tested in isolation:

1. **Rules are calibrated.** The v0.18.9 v8.5 calibration on 546,258 files shows the rules fire reliably (e.g. `dead/unused-import` has 11,148 TP / 5,586 FP, 30.06x lift). This is a property of the *detector*, not of any consumer of the detector.

2. **The tool returns the structure.** `slop_suggest_with_structure` returns either the cached `.slopbrick/structure.md` (fast path) or the live re-scan (slow path, annotated with `structureHint`). The fast-path text is the same markdown an agent would see if it called `slopbrick scan` and read the output. This is a property of the *transport*, not of the agent.

3. **The agent produces code.** This is the property we don't test. Does an agent that *receives* the structure markdown actually write code that matches the structure? A 30x-lift rule is worthless if the agent ignores it and writes `useState` in a server component anyway.

VibeDrift's published eval (the "sharpest gap" reference from the strategic review) measures layer 3 directly: they run their tool against a fixed set of coding tasks in fixed codebases, with and without the tool, and report the conformance delta. The eval is the evidence that the tool *changes agent behavior*, not just that the tool *can produce a correct answer*.

---

## The hypothesis

**H1:** An AI agent that has access to `slop_suggest_with_structure` produces code that conforms to the codebase's structure at a higher rate than the same agent without the tool.

**H0 (null):** The tool has no measurable effect on agent behavior. Agents write the same code either way, and the structure information is ignored.

The eval is designed to reject H0 if the conformance delta is statistically significant. The minimum detectable effect is set by the test-case count and the per-case variance; with ~30 test cases and binary conformance scoring, we can detect a delta of ~20 percentage points at 80% power and α=0.05.

## The eval design

### Test cases

A test case is a triple:

```typescript
type TestCase = {
  id: string;                          // e.g. "react-server-component-01"
  codebase: {                          // the codebase the agent starts in
    path: string;                      // absolute path to a fixture
    description: string;               // what the codebase is
  };
  task: {                              // the coding task spec
    description: string;               // what the agent should do
    targetFiles: string[];             // which files the agent will edit
  };
  groundTruth: {                       // the structure the agent should conform to
    allowedPatterns: string[];         // patterns the agent may use
    deniedPatterns: string[];          // patterns the agent must avoid
    conventions: string[];              // naming, structure, etc.
  };
};
```

**Initial seed: 3 test cases.** Each is a minimal codebase (one component file + one test file + one `package.json`) with a clear structure. The agent's task is "add a new feature" or "refactor an existing component." The ground truth is derived from the codebase's own `.slopbrick/constitution.json` (if present) or from a hand-written spec.

### Agent setup

The agent is a function:

```typescript
type Agent = (task: TestCase, tools: ToolConfig) => Promise<AgentOutput>;
type ToolConfig = { slop_suggest_with_structure: boolean };
type AgentOutput = { files: Record<string, string> /* path -> content */ };
```

The `ToolConfig` has two settings: with the tool, without the tool. The control (without) is the same agent with `ToolConfig.slop_suggest_with_structure: false` — the agent is given the codebase and the task, nothing else.

The treatment (with) is the same agent with `ToolConfig.slop_suggest_with_structure: true` — the agent can call the tool and receive the structure markdown before making changes.

**For the v0.22 stub, the agent is a deterministic function that returns a fixed output.** The stub does not call an LLM. It exists so the conformance check, the test-case loader, and the comparison framework can be validated end-to-end. The LLM wiring is a separate PR.

### Conformance metric

Conformance is a binary score per test case: did the agent's output conform to the ground truth?

```typescript
type ConformanceResult = {
  taskId: string;
  toolEnabled: boolean;
  precision: number;   // of the agent's changes, % that match allowedPatterns
  recall: number;      // of the deniedPatterns, % that the agent avoided
  f1: number;          // harmonic mean of precision and recall
  conformant: boolean; // f1 >= 0.8 (the pass/fail threshold)
};
```

The metric is computed by:
1. Extracting the diff between the agent's output and the original codebase
2. Running `slopbrick scan` on the agent's output (with rules matching the ground truth)
3. Comparing the scan findings to the ground truth (allowedPatterns / deniedPatterns)
4. Computing precision (true positives / total agent changes) and recall (true negatives / total denied patterns)

### Comparison

The eval runs each test case twice: once with the tool, once without. The result is a per-task delta:

```typescript
type EvalResult = {
  taskId: string;
  withTool: ConformanceResult;
  withoutTool: ConformanceResult;
  delta: { precision: number; recall: number; f1: number; conformant: boolean };
};
```

The aggregate result is the mean delta across all test cases, with a 95% confidence interval. The eval rejects H0 if the mean f1 delta is significantly positive (one-sided t-test, α=0.05).

## What v0.22 ships

The v0.22 release ships three things:

1. **The harness** (`tests/research/controlled-eval.test.ts`) — the test framework, the conformance check, the agent stub, and 3 seed test cases. This is what this design doc describes. The harness runs as a vitest test suite and produces a JSON report.

2. **The agent stub** — a deterministic function that returns a fixed output for each test case. The stub exists so the harness is testable without an LLM dependency. The stub is documented as "to be replaced with a real LLM call in a follow-up."

3. **The report format** — a Markdown report that summarizes the per-task deltas, the aggregate mean delta, and the H0/H1 verdict. The report is the artifact that proves the tool changes agent behavior.

What v0.22 does **not** ship:
- A real LLM integration. The harness is LLM-agnostic; the user wires their own model (Claude, GPT-4, local) via a single function in the agent stub.
- A published conformance number. The v0.22 numbers are from the stub, not a real LLM. The first real-LLM numbers ship in v0.23 or later.

---

## Why the harness is LLM-agnostic

The eval's value is in the *test cases* and the *conformance metric*, not in which LLM is used. By keeping the agent a function with a `ToolConfig` parameter, the harness can be run with any LLM (or any non-LLM code generator) without changing the test cases or the metric. This is important because:
- Different LLMs will produce different conformance numbers, and the harness should be able to report all of them.
- The LLM landscape changes faster than the eval design; we don't want to redo the eval every time a new model is released.
- A "code-generating non-LLM" (e.g., a deterministic refactoring tool) can be evaluated with the same harness.

The agent stub is a placeholder that returns a fixed output. Replacing it with a real LLM is a one-line change in `tests/research/controlled-eval.test.ts`:

```typescript
// Before (stub):
const agent: Agent = async (task) => stubOutput(task);

// After (real LLM):
const agent: Agent = async (task, tools) => llmAgent(task, tools);
```

## What changes when the LLM is wired

Three things:
1. `stubAgent` is replaced with a real LLM call (Claude, GPT-4, local, etc.).
2. The agent's tool-calling protocol is implemented (the LLM must know how to invoke the MCP tool).
3. The harness produces a real conformance number per test case, and the aggregate mean delta is the headline metric.

The harness design is intentionally LLM-agnostic so the LLM choice is a configuration decision, not a code change.

## The VibeDrift comparison

VibeDrift's published eval (the reference from the strategic review) measures the same property — whether the tool changes agent behavior — but uses a different methodology:
- They run the eval on a public benchmark of coding tasks (e.g., SWE-bench).
- They report per-task pass rates with and without the tool.
- They publish the aggregate number on their website.

The v0.22 design differs in two ways:
1. **The codebase is the test case, not the task.** VibeDrift's test cases are individual tasks (one per case). Ours are codebases (the agent works *in* a codebase, not *on* a task in isolation). This is a more realistic measure of conformance because real agents work in codebases, not in isolation.
2. **The conformance metric is rule-based, not pass/fail.** VibeDrift's metric is binary (the task passes or it doesn't). Ours is a precision/recall F1 against a ground-truth structure spec. This is more informative because it tells us *which* rules the agent is failing to follow, not just that the task failed.

The first version of the VibeDrift comparison is a side-by-side report in the v0.22 release notes: slopbrick's number, VibeDrift's number, and the methodology differences. The user can decide which is more relevant for their use case.

## Open questions

1. **Test case selection.** How do we pick the initial 3 test cases? Options: (a) hand-pick from the existing `tests/fixtures/` directory, (b) generate from the v8.5 corpus, (c) take from a public benchmark. Option (a) is the fastest path to v0.22.

2. **Conformance threshold.** The current design uses f1 ≥ 0.8 as the pass/fail threshold. This is arbitrary. The v0.22 numbers should include a threshold sensitivity analysis (what if the threshold is 0.7? 0.9?).

3. **LLM cost.** A real-LLM eval with 3 test cases × 2 conditions = 6 runs. At ~$0.10 per run, that's $0.60 per eval. A 30-test-case eval is $6. The v0.22 stub avoids this cost; the v0.23+ real-LLM evals need a budget.

4. **Tool-call protocol.** The MCP tool returns the structure as a text block. The agent (LLM) must know to call the tool before writing code. This is a prompt-engineering question: how do we instruct the LLM to use the tool? The v0.22 stub doesn't address this; the v0.23 work does.

## Next steps

- **v0.22 (this release):** ship the harness stub + 3 seed test cases + the report format. No LLM.
- **v0.23:** wire a real LLM (Claude or GPT-4) to the agent stub. Run the eval end-to-end. Publish the conformance delta.
- **v0.24:** expand to 10–30 test cases. Add the threshold sensitivity analysis. Add the VibeDrift side-by-side comparison.
- **Long-term:** the eval becomes a CI gate. Every release of `slop_suggest_with_structure` is validated against the eval; a regression in conformance is a release blocker.

The v0.22 harness is the foundation. The real numbers come later.
