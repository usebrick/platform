# v10.3 scanner and feature-schema hash contract

**Recorded:** 2026-07-13
**Status:** design/evidence only; not an implementation or admission approval

This note closes the audit question around the scanner-code and feature-schema
hashes named by the v0.45 continuation plan. It deliberately does not add
manifest fields or claim that any current build is an authoritative calibration
artifact.

## Current evidence and decision

The v10.3 run manifest currently binds the registry, signal table, config,
corpus manifest, selection, and local checkout map. The Core runtime validator,
JSON Schema, generated declaration, CLI `run:init`, `scan`, `verify`, and resume
path all agree on that six-hash contract. None of those paths binds scanner
bytes or a feature-schema byte set.

The calibration worker is produced by the package build and can resolve a
generated worker bundle. The source worker and generated bundle are separate
representations, and the current contract does not bind the build receipt,
the package/dependency closure, or a clean commit. Hashing whichever file is
present would therefore make source-versus-build selection a provenance gap.

The research feature names are duplicated in two Python scripts. There is no
versioned schema consumed by both extraction and runtime, no ordered value/missing
value contract, and no immutable feature artifact to hash. Those scripts and
their output remain exploratory and cannot authorize calibration.

**Decision:** do not add placeholder `scannerSha256` or `featureSchemaSha256`
fields to the current v10.3 contract. Do not hash a caller-selected source or
`dist` file, and do not make a legacy run appear stronger by filling either
field with a package, Git, or source-tree hash.

## Canonical contract to implement after the authority boundary is ready

The next implementation should introduce a new method revision (proposed
`v10.3.2`). Legacy `v10.3.0` and `v10.3.1` manifests remain readable and retain
their current six-hash semantics. A `v10.3.2` manifest must contain these two
additional lowercase SHA-256 values under `inputHashes`:

```text
scannerArtifactSha256
featureSchemaSha256
```

### Scanner artifact

First freeze a clean, reproducible package boundary. The scanner authority is a
canonical, path-free JSON receipt (for example, `scanner-artifact-v1.json`) that
contains:

- the exact package/tarball and clean builder identity;
- the scanner entrypoint relative name and its byte SHA-256;
- every bundled runtime file and every external runtime/native dependency file
  that can affect the worker, each with a relative name, byte length, and SHA-256;
- the supported Node and package-manager policy;
- the build recipe and artifact format version.

The receipt is canonical JSON (sorted keys, stable arrays, UTF-8, one trailing
newline) and contains no checkout or user-home paths. Its own SHA-256 is the
`scannerArtifactSha256` value. A consumer must reopen the receipt, rehash the
listed bytes, verify the clean builder/tarball binding, and only then invoke the
worker. A missing, extra, or changed runtime byte fails closed. This makes the
hash identify the complete scanner behavior rather than whichever source or
generated file happened to be found first.

The artifact producer must also remove the source/generated fallback ambiguity:
calibration execution accepts only the receipt-selected entrypoint. The normal
package CLI may keep its existing development fallback, but that fallback must
be rejected by the v10.3.2 calibration invoker.

### Feature schema

Create one versioned JSON Schema (proposed
`schemas/v1/calibration-feature.schema.json`) and a generated TypeScript type.
The schema must specify the feature-schema version, ordered feature names, value
types and finite ranges, missing-value semantics, and the file/repository/family
identifiers carried with each row. The Python extractor, any TypeScript
producer, and the validator all read this same file; duplicated hard-coded
feature-name lists are removed or checked against it.

`featureSchemaSha256` is the SHA-256 of the canonical schema bytes, not of a
Python source file, a generated JSONL dataset, or a model output. A feature row
must carry the schema version and the run's two artifact hashes so a dataset
cannot silently mix feature definitions or scanner revisions.

## Schema and CLI migration

Implement this only after the two authority artifacts and their receipts exist:

1. Extend the Core run-manifest schema and generated declaration with the two
   hash fields, preserving legacy fixtures and rejecting unknown fields. The
   semantic validator requires them exactly when `methodVersion` is
   `v10.3.2`; legacy methods remain valid without them.
2. Extend `RunManifestDraft` and `createV103RunManifest` to accept the two
   verified hashes only for `v10.3.2`. Compute them from the validated receipts;
   never trust a caller-provided digest without rehashing bytes.
3. Add explicit calibration-only inputs (artifact receipt and feature-schema
   receipt). Do not put local paths into `commandArgs`; record stable artifact
   IDs/tokens instead. `run:init` must verify both receipts before writing the
   manifest.
4. Reopen and rehash the same receipts in `scan`, `verify`, `cal:report`, and
   resume. Compare the complete frozen manifest before scanner invocation or
   any output mutation. The existing canonical run-manifest hash then binds
   durable attempts transitively.
5. Make the v10.3.2 worker invoker require the receipt-selected entrypoint and
   reject source or generated fallback resolution. Keep v10.3.0/v10.3.1 paths
   unchanged for historical compatibility.

## Required TDD and verification matrix

The implementation is not ready for a green claim until these tests exist and
fail before the corresponding code is written:

- Core schema/runtime fixtures: valid v10.3.2, missing scanner hash, missing
  feature hash, malformed hash, extra input field, and unchanged legacy
  v10.3.0/v10.3.1 fixtures.
- Receipt canonicalization: key/order and newline determinism; path rejection;
  missing/extra/changed runtime byte rejection; tarball/clean-builder mismatch;
  dependency/native-byte mismatch; and receipt self-hash mismatch.
- CLI `run:init`, `scan`, `verify`, `cal:report`, and `--resume`: each rejects
  a changed scanner or feature receipt before invoking the scanner or writing
  final artifacts. Resume must reject a stale completed attempt under the new
  manifest hash.
- Worker selection: v10.3.2 accepts only the receipt-selected artifact and
  proves that source/generated fallback is rejected; output remains path-free.
- Feature producer parity: Python and TypeScript consumers read the same schema,
  preserve order and missing-value semantics, and reject an unknown schema
  version. Dataset rows include both frozen hashes.
- Packed-consumer proof: clean installed package, Node 22 and Node 24, offline
  receipt validation and scan/resume/verify; record the exact tarball and
  receipt hashes. This is separate from the current dirty-tree evidence.

## Gate order and stop conditions

1. **Authority lock:** scanner owner approves the receipt format and a clean
   builder produces the exact package artifact. No manifest output yet.
2. **Feature lock:** data/statistical owner approves one schema and extractor
   parity; research scripts are migrated or explicitly quarantined.
3. **Core/CLI migration:** schema, generated types, semantic validator, and the
   TDD matrix above pass on both supported Node versions.
4. **Deterministic smoke:** two independent 100/100 runs agree after timestamp
   normalization and prove receipt/hash binding.
5. **Admission/canary:** only after provenance, coverage, and leakage gates pass
   may the exact 5,000/5,000 canary and full calibration consume the artifacts.

Until these gates close, scanner/feature hash fields remain an open plan item;
no corpus label, metric, rule verdict, release, tag, publish, or deployment may
be promoted from a source/dist fallback or an exploratory feature file.
