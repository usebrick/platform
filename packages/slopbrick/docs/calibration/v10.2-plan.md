#### Problem 2: The calibrator hangs on most chunks

The calibrator's `runScan()` loop uses `execFileSync` to spawn
`slopbrick scan` as a child process per 600-file chunk. Empirical
behavior:

| Calibration run | Result |
|-----------------|--------|
| `--positive-limit 50 --negative-limit 50` | ✅ 16s, 25 rules |
| `--positive-limit 600 --negative-limit 600` | ✅ 87s, 65 rules |
| `--positive-limit 3000 --negative-limit 3000` | ✅ 7m33s, 75 rules |
| `--positive-list {all}` full corpus | ❌ 14+ min hang after chunk 1 |
| Manual `slopbrick scan` of first 600 positive | ✅ 35s |
| Manual `slopbrick scan` of first 600 negative | ✅ 13s |

The chunk-by-chunk timing is fine in isolation (35s positive, 13s
negative). But the orchestrator's `execFileSync` loop hangs after the
first chunk. The hang is NOT in the scan child — it's in the parent
calibrator waiting for microtasks to drain after a child exits. The
`try { execFileSync(...) } catch {}` swallows child errors but leaves
the parent's event loop in an unrecoverable state.

Manual scan of any 600-file slice via `slopbrick scan` works fine,
proving the hang is in the spawn-and-wait loop, NOT in the scanner.

**v4 race-condition fix (2026-07-09):** After the initial v4 run, audit
revealed pos-elasticsearch-files.txt was 0 bytes despite 30k+ Java
files being in the corpus. Root cause: the script's `discover_repos`
returned the repo **basename** ("elasticsearch") and then a separate
`resolve_repo_dir` looked it back up by name with
`find -maxdepth 4 -name elasticsearch | head -1`. The first match was
a subdirectory `positive/elasticsearch/docs/reference/elasticsearch`
(zero Java files), not the real repo root.

**Fix**: `discover_repos` now emits the full repo path (parent of
`.git`) directly. The script then derives the basename for the
output filename. This eliminates the race entirely.

After the fix, the corpus jumped from 466k to **622,550 files** (+33%
on top of the v4 baseline). The biggest single-repo gain was
elasticsearch (0 → 30,565). Other repos with subdirectories of the
same name likely had similar but smaller corruptions — collectively
those explain the extra 156k files.

### Phase 9 — Parallel chunk scan via xargs (bypass the spawn loop) 🔄 NEW

The calibrator's spawn loop is buggy at scale. Workaround: bypass it
entirely using a parallel xargs pipeline + a tiny merge script.
**Action** (1 hr):

```bash
# Split filelists into 600-file chunks
mkdir -p /tmp/cal-chunks/pos /tmp/cal-chunks/neg
split -l 600 /Users/cheng/corpus-expansion/filelists/pos-all-files.txt /tmp/cal-chunks/pos/chunk-
split -l 600 /Users/cheng/corpus-expansion/filelists/neg-all-files.txt /tmp/cal-chunks/neg/chunk-

# Run 4 parallel scan workers per polarity, writing per-chunk JSONs
cat /tmp/cal-chunks/pos/chunk-* | xargs -n 1 -P 4 -I {} \
  bash -c 'node bin/slopbrick.js scan $(cat {}) --json {}.json --no-telemetry --quiet'

cat /tmp/cal-chunks/neg/chunk-* | xargs -n 1 -P 4 -I {} \
  bash -c 'node bin/slopbrick.js scan $(cat {}) --json {}.json --no-telemetry --quiet'
```

Then a small TS script that reads all per-chunk JSONs and computes
the calibration report the same way `calibrate.ts` does.

**Implemented in**: `packages/slopbrick/scripts/cal/scan-parallel.sh`
and `packages/slopbrick/scripts/cal/merge-chunk-results.ts`.

### Quality review of v10.2 results (2026-07-09)

The v10.2 calibration run produced surprising results: 55% of rules
(57/104) had <33% precision, including many that v10.1 had calibrated
as "strong" (precision ≥ 65%). A focused review revealed two
distinct problems that require recalibration before any
signal-strength.json update.

**Problem 1 — Test-file inflation dominates the inversions.**
The v4 corpus includes `tests/`, `__tests__/`, etc. (v3 excluded
them). This was right for "more data is better" but wrong for
calibration because test files naturally contain patterns that
fire AI/quality rules (mocks, weak assertions, console.log debug)
but aren't AI-specific.

| Pattern | Pos fires | Neg fires | Neg fires in tests |
|---|---:|---:|---:|
| `dup/identical-block` | 122,785 | 396,657 | majority (test fixtures) |
| `dead/unused-local` | 1,743 | 88,214 | majority (test mocks) |
| `ai/compression-profile` | 35,175 | 56,771 | high |

**Problem 2 — Per-language corpus imbalances** make several rules'
"USEFUL" verdicts meaningless:
- `.cs` (C#): 99% pos / 1% neg — no real C# in neg corpus
- `.swift`: 98% pos / 2% neg
- `.go`: 95% pos / 5% neg
- `.kt`: 83% pos / 17% neg — 100% prec is small-sample artifact
- `.rb`: 0% pos / 99% neg — Ruby AI corpus is empty

**Decision: DO NOT apply v10.2 verdicts as-is.** Two recalibration
passes are required:

**Pass A (v10.2a)**: Re-run with tests/ excluded for the AI/quality
rules. This validates whether v10.1's verdicts were right and the
v10.2 inversions are a test-inclusion artifact. If `ai/compression-profile`
reverts to ~75% precision, v10.1 was right and the test files are
the problem. If it stays at ~38%, the rule itself is too broad and
needs tightening.

**Pass B (v10.2b)**: Re-run with `--include-rule` to scan ONLY rules
with balanced languages (≥1000 files in BOTH polarities). This skips
language-specific rules whose verdicts are corpus-gap artifacts.

**Pass C (apply)**: After Pass A and B:
1. Rules inverted in BOTH v10.1 AND v10.2: mark INVERTED/defaultOff=true
   (stable inversions: `dead/unused-parameter`, `dead/unreachable`,
   `logic/math-variable-name-entropy`, `cpp/printf-debug`, etc.)
