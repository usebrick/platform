## Context

The slopbrick calibration pipeline runs positive (AI-generated) and
negative (real human) corpora through the rule engine and computes
per-rule precision, recall, FPR, and lift. The `calibrate`
subcommand in `packages/slopbrick/src/cli/commands/calibrate.ts`
drives this end-to-end.

## Phase 1: Add missing language rules (4 hrs, low risk)

Four language rule directories need to be created. Priority order
matches corpus coverage (more files = better calibration signal).

### 1a. Kotlin (`kt/`) — 373 positive files

- `kt/string-template-injection` — `"$name" in SQL` is a
  Kotlin equivalent of Java's `+ name + ` SQL concat. The
  `kotlin/sql-string-concat` rule is Kotlin's analog for SQL;
  this rule covers the same anti-pattern in HTML/server-side
  templating (Ktor, kotlinx.html).
- `kt/coroutine-cancellation-missing` — `launch { someWork() }`
  without `ensureActive()` or `cancellable` is a leak. The
  Kotlin coding conventions doc explicitly calls out that every
  coroutine must be cancellable.
- `kt/force-unwrap` — mirror `swift/force-unwrap` and
  `rust/unwrap-in-production`. The `!!` operator bypasses null
  safety, the same way `!` does in Swift. Per the Kotlin
  convention: "Avoid using the not-null assertion operator (`!!`)"
- `kt/global-coroutine-scope` — using `GlobalScope.launch { }`
  is an anti-pattern that defeats structured concurrency. The
  Kotlin docs (kotlinlang.org) explicitly call this out.

### 1b. Ruby (`rb/`) — 12,680 negative files

- `rb/sql-string-concat` — `"SELECT * FROM users WHERE id = " + id`
  is Ruby's analog of Java's SQL concat. `ActiveRecord` exists
  for this reason. Per the Ruby style guide, the right tool is
  parameterized queries (`User.where(id: id)`).
- `rb/exception-swallowing` — `rescue => e; nil` (or just
  `rescue; nil`) is a smell. The Ruby style guide says rescues
  should be narrow and re-raise or log.
- `rb/n-plus-one-query` — `.each { |x| x.children.each { ... } }`
  pattern in ActiveRecord is N+1. Ruby's `includes` solves this.

### 1c. C# (`cs/`) — 6,530 positive files

- `cs/sql-string-interpolation` — `$"SELECT * FROM users WHERE id
  = {userId}"` is the C# interpolation-string SQL injection
  variant. EF Core's `FromSqlInterpolated` solves it.
- `cs/async-without-await` — `async Task DoStuff() { SomeCall(); }`
  runs synchronously. Per the Microsoft docs: "Marking a method
  as async without await inside it is a code smell."
- `cs/empty-catch-block` — `catch { }` or `catch (Exception) { }`
  swallows errors. The Microsoft coding conventions call this
  out explicitly.

### 1d. PHP (`php/`) — 152 files

- `php/sql-injection` — `"SELECT * FROM users WHERE id = " . $_GET['id']`
  is the PHP equivalent of Java's SQL concat. The fix is
  prepared statements via PDO. The PSR-12 style guide doesn't
  cover security directly, but OWASP does.
- `php/empty-catch` — `catch (Exception $e) {}` swallows errors.
  Different from C# empty-catch because PHP's `catch` doesn't
  require an exception type.

### Rule file template

Each rule follows the pattern in `packages/slopbrick/src/rules/java/sql-string-concat.ts`:

1. **JSDoc header**: rule id, category, severity, aiSpecific
2. **"Why this matters"** section citing OWASP / language docs
3. **Calibration status**: DORMANT until v10.2 corpus run
4. **Scope**: file-local, regex/ast pattern matching
5. **Falsifiability**: the test cases that should NOT fire

- `packages/slopbrick/src/rules/swift/force-unwrap.ts` — Swift mirror

## Phase 3: Per-language calibration filter (2 hrs, medium risk)

**Current problem:** The calibrator scans ALL files for ALL rules.
`cpp/c-style-cast` gets evaluated on Python files, which inflates
FPR. The calibration is not language-aware.

**Fix:** Modify `packages/slopbrick/src/research/calibrator.ts` to
add a language filter:

```typescript
// In runScan, before passing the file to the slopbrick CLI:
const lang = detectLanguage(filePath);
const applicableRules = rules.filter(r => 
  !r.language || r.language === lang
);
```

Where each rule declares its language (or "any"). The `Language` enum
matches the rule categories: `cpp`, `java`, `go`, `rust`, `swift`,
`ts`, `kt` (kotlin), `rb` (ruby), `cs` (csharp), `php`.

**Reference:** arXiv:2409.01382 (Rahman et al. 2024) — function-level
and class-level detection rely on fundamentally disjoint structural
signatures. This implies per-language calibration is necessary for
accurate precision/recall, not just an optimization.

## Phase 6: Make flywheel non-determinism predictable (4 hrs, high risk)

**Current problem:** The flywheel auto-tunes on every scan, so the
same workspace gives different results between runs. The
`c59de2a50` fix made the counts accurate but the distribution
still varies.

**Options (pick one):**

**(a) Make auto-tuning opt-in:**
```bash
slopbrick scan --tune   # new flag to run auto-tuning
slopbrick scan          # default: no auto-tuning, stable
```

**(b) Run auto-tuning on a schedule:**
- Auto-tuning runs once per day (CI cron job)
- Stored in a separate file, applied as patches
- Stable scans don't re-tune

**(c) Add `--stable` mode:**
```bash
slopbrick scan --stable  # skip flywheel entirely
```

**Recommendation:** Option (a) — make tuning explicit. Power users
can tune; CI runs are stable by default.

**Files to change:**
- `packages/slopbrick/src/engine/flywheel.ts` — gate auto-tune
- `packages/slopbrick/src/cli/commands/scan.ts` — add `--tune` flag
- `packages/slopbrick/src/cli/report/persistRun.ts` — only persist
  tuned state when `--tune` is set
