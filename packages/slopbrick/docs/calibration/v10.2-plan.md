## Context

The slopbrick calibration pipeline runs positive (AI-generated) and
negative (real human) corpora through the rule engine and computes
per-rule precision, recall, FPR, and lift. The `calibrate`
subcommand in `packages/slopbrick/src/cli/commands/calibrate.ts`
drives this end-to-end.

#### Problem 2: The calibrator hangs on most chunks

The calibrator's `runScan()` loop uses `execFileSync` to spawn
`slopbrick scan` as a child process per 600-file chunk. Empirical
behavior:

| Calibration run | Result |
|-----------------|--------|
| `--positive-limit 50 --negative-limit 50` | ✅ 16s, 25 rules |
| `--positive-limit 600 --negative-limit 600` | ✅ 87s, 65 rules |
| `--positive-limit 3000 --negative-limit 3000` | ✅ 7m33s, 75 rules |
| `--positive-list {all}` full corpus | ❌ 14+ min hang after chunk 1 |
| Manual `slopbrick scan` of first 600 positive | ✅ 35s |
| Manual `slopbrick scan` of first 600 negative | ✅ 13s |

The chunk-by-chunk timing is fine in isolation (35s positive, 13s
negative). But the orchestrator's `execFileSync` loop hangs after the
first chunk. The hang is NOT in the scan child — it's in the parent
calibrator waiting for microtasks to drain after a child exits. The
`try { execFileSync(...) } catch {}` swallows child errors but leaves
the parent's event loop in an unrecoverable state.

Manual scan of any 600-file slice via `slopbrick scan` works fine,
proving the hang is in the spawn-and-wait loop, NOT in the scanner.

### Baseline (sanity check) — v0.10.1 v10.2-3k.md

The 3k+3k subset calibration (completed in 7m33s) produced:

- 75 rules evaluated (out of 115)
- 42 strong, 12 weak, 21 inverted
- Notable inversions at the smaller sample:
  - `ai/compression-profile` precision 60% (was weak at v0.10.1 too)
  - `ai/comment-ratio` precision 38.7% (INVERTED at this sample)
  - `ai/errors-near-eof` precision 40.6% (INVERTED)
- The new language rules (kt/cs/rb/php) didn't fire because no files
  of those extensions were in the 3k+3k sample.

This smaller calibration is **statistically valid but biased**: it's a
sample of the easy cases (TS/JS/Python/Go). The hard cases (Rust,
Java, C++) and the new language rules need their own sample.

### Phase 4 — Per-chunk timeout + pathological-file skip 🔄 NEW, needed

The calibrator's `runScan()` loop uses `execFileSync` with no timeout.
A pathological file (e.g. a 50MB CommonJS file with deeply nested
array literals, or a minified single line 1MB+ JS) can take 5-20 min
to parse without timing out, leaving the orchestrator stuck.

**Action** (2 hrs):
- Add per-chunk timeout (default 90s, configurable via `--chunk-timeout`)
- If a chunk times out, log + skip + continue
- Track skipped chunks in the calibration report

### Phase 5 — Update DORMANT comments to v10.2 🟡 PARTIAL

  corpus calibration (1.13M files)" instead of stale "v10 corpus
  calibration"
- 12 new language rule comments already say "v10.2 corpus
  calibration"

Commit `f6eb98f75`.

### Phase 6 — Make flywheel `--tune` opt-in ❌ DEFERRED

Currently the flywheel auto-tunes rule severities during `scan`.
This is the source of the "flywheel non-determinism: same scan
gives different results between runs" issue. Making it opt-in
(behind `--tune`) would give reproducible scans.

**Risk:** HIGH. The flywheel has been silently changing rule
severities across users' projects. Changing default behavior
could break reproducibility for users who rely on flywheel-tuned
severities.

**Decision:** Defer to v0.44. Phase 2 calibration is more important
for v0.43.0 → v0.10.2.

### Phase 9 — Parallel chunk scan via xargs (bypass the spawn loop) 🔄 NEW

The calbrator's spawn loop is buggy at scale. Workaround: bypass it
entirely using a parallel xargs pipeline + a tiny merge script.

**Action** (1 hr):

```bash
# Split filelists into 600-file chunks
mkdir -p /tmp/cal-chunks/pos /tmp/cal-chunks/neg
split -l 600 /Users/cheng/corpus-expansion/filelists/pos-all-files.txt /tmp/cal-chunks/pos/chunk-
split -l 600 /Users/cheng/corpus-expansion/filelists/neg-all-files.txt /tmp/cal-chunks/neg/chunk-

# Run 4 parallel scan workers per polarity, writing per-chunk JSONs
cat /tmp/cal-chunks/pos/chunk-* | xargs -n 1 -P 4 -I {} \
  bash -c 'node bin/slopbrick.js scan $(cat {}) --json {}.json --no-telemetry --quiet'

cat /tmp/cal-chunks/neg/chunk-* | xargs -n 1 -P 4 -I {} \
  bash -c 'node bin/slopbrick.js scan $(cat {}) --json {}.json --no-telemetry --quiet'
```

Then a small TS script that reads all per-chunk JSONs and computes
the calibration report the same way `calibrate.ts` does.
