# v10.3 immutable release-asset materialization implementation plan

> **For the implementing agent:** REQUIRED SKILLS: use
> `superpowers:test-driven-development`,
> `superpowers:subagent-driven-development`, and
> `superpowers:verification-before-completion`. Execute one task at a time,
> request an independent review after every scoped commit, and do not promote a
> corpus merely because materialization succeeded.

**Goal:** Extend the v10.3 calibration source contract so reviewed files may
come from an immutable, checksum-pinned HTTPS release archive without claiming
that those files exist in a Git tree.

**Architecture:** Keep the existing Git-tree repository record as the
backward-compatible default. Add one explicit `release_archive` materialization
variant whose archive bytes, extraction policy, and repository-relative root
are hashed into canonical source identity. A separate intake command downloads
and safely extracts archives into a content-addressed cache, then emits a
local-only checkout map. Selection and scanning remain path-free; the resolver
accepts only a checkout-map entry whose artifact binding exactly matches the
selected record and still verifies each file's SHA-256.

**Technology:** TypeScript, Node 20+/22/24 built-ins, JSON Schema draft
2020-12, AJV, Vitest, `yauzl@3.4.0` for bounded streaming ZIP reads,
`yazl@3.3.1` as a test-only fixture writer, matching `@types` packages, and
the existing canonical JSON/SHA-256 helpers.

**Primary-source basis:** `yauzl` documents central-directory parsing,
asynchronous bounded-memory reads, unsafe-name validation, and entry-size
validation; those are useful primitives, not the complete policy. SlopBrick
must still enforce its own path, type, duplicate, resource, redirect, cache,
and atomicity rules. GitHub release URLs remain locators: the recorded archive
SHA-256, not the URL or tag, is the immutable identity.

**Execution mode:** Existing-codebase `team-pr` discipline on the current
`codex/v0.45-recovery` branch: one task-owning implementer at a time, scoped
Conventional Commits, then a fresh read-only reviewer. Preserve every unrelated
dirty/untracked user file, stage only named task files, use `corepack pnpm`, and
do not push, tag, publish, deploy, or change remote/repository settings.

## Frozen decisions and non-goals

1. An omitted repository `materialization` means the current Git-tree behavior.
   Existing v10.3.0 Git manifests remain valid and keep their current
   `sourceId` and `fileId` values.
2. A release archive requires method version `v10.3.1` or later. It does not
   bump `STRUCTURE_SCHEMA_VERSION`; this is an additive calibration-only
   contract.
3. The repository record remains tied to an immutable upstream Git commit for
   project/license context, while `materialization.assetSha256` identifies the
   actual archive bytes that contain the file.
4. Canonical release-archive source IDs use this exact form:

   ```text
   <repositoryId>@<commitSha>+asset-<assetSha256>:<normalizedPath>
   ```

5. The manifest's `normalizedPath` is relative to
   `materialization.rootPrefix`, never to the host cache. The emitted checkout
   map points at that contained root.
6. The downloader/extractor never runs inside the resolver. `scan` has no
   network access and consumes only an already verified materialization.
7. Local files, `file:` URLs, mutable/unhashed assets, unchecked redirects,
   and user-supplied extracted directories remain inadmissible.
8. This work does not decide labels, tiers, splits, duplicate safety, license
   compatibility, or release eligibility. Those remain separate provenance
   and calibration-review gates.
9. This work supports ZIP only. Tar, Git LFS, OCI, S3, model hubs, and generic
   generated-directory imports require new policy versions and separate
   threat-model reviews.

## Exact contract shape

The optional repository field is:

```ts
interface ReleaseArchiveMaterialization {
  readonly kind: 'release_archive';
  readonly assetUrl: string;             // absolute HTTPS URL
  readonly assetSha256: string;          // lowercase 64-hex
  readonly assetBytes: number;           // exact positive safe integer
  readonly archiveFormat: 'zip';
  readonly rootPrefix: string;            // safe archive-relative directory
  readonly extractionPolicy: 'safe-zip-v1';
}
```

The optional local checkout-map binding is deliberately smaller:

```ts
interface ReleaseArchiveCheckoutBinding {
  readonly kind: 'release_archive';
  readonly assetSha256: string;
  readonly extractionPolicy: 'safe-zip-v1';
}
```

Selection records copy that binding. They do not copy `assetUrl`,
`assetBytes`, or a host path; the copied canonical manifest and its hash retain
the full acquisition record.

`safe-zip-v1` is a frozen algorithm, not a prose label. It must enforce:

- at most 100,000 entries;
- at most 1 GiB total declared uncompressed bytes;
- at most 32 MiB per regular file;
- at most 200:1 declared uncompressed/compressed ratio per file and in total;
- path length at most 4,096 characters and depth at most 64 segments;
- UTF-8 names with `/` separators only;
- no NUL, backslash, absolute path, drive prefix, empty interior segment,
  `.` segment, or `..` segment;
- no duplicate normalized path and no Unicode-normalized/case-folded path
  collision;
- regular files and directories only; reject symlinks, devices, FIFOs,
  sockets, encrypted entries, and unsupported compression methods;
- an empty temporary extraction directory, `wx` file creation, fixed local
  permissions, ignored archive mtimes/permissions, and atomic final rename;
- a deterministic receipt/inventory that is fully reverified before cache
  reuse.

Changing any limit or rule creates `safe-zip-v2` and a new calibration method
version. Do not silently loosen `safe-zip-v1` after observing a corpus.
