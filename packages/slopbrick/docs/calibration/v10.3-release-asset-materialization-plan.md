**Goal:** Extend the v10.3 calibration source contract so reviewed files may
come from an immutable, checksum-pinned HTTPS release archive without claiming
that those files exist in a Git tree.

**Architecture:** Keep the existing Git-tree repository record as the
backward-compatible default. Add one explicit `release_archive` materialization
variant whose archive bytes, extraction policy, and repository-relative root
are hashed into canonical source identity. A separate intake command downloads
and safely extracts archives into a content-addressed cache, then emits a
local-only checkout map. Selection and scanning remain path-free; the resolver
accepts only a checkout-map entry whose artifact binding exactly matches the
selected record and still verifies each file's SHA-256.

**Technology:** TypeScript, Node 20+/22/24 built-ins, JSON Schema draft
2020-12, AJV, Vitest, `yauzl@3.4.0` for bounded streaming ZIP reads,
`yazl@3.3.1` as a test-only fixture writer, matching `@types` packages, and
the existing canonical JSON/SHA-256 helpers.

**Primary-source basis:** `yauzl` documents central-directory parsing,
asynchronous reads, raw names, custom random-access readers, and entry-size
validation; those are useful primitives, not the complete policy. Its decoded
name validation, local-header behavior, and missing CRC check do not satisfy
`safe-zip-v1`, so SlopBrick binds a raw parser and borrowed descriptor reader to
its own path, type, duplicate, resource, cache, and publication rules. GitHub
release URLs remain locators: the recorded archive SHA-256, not the URL or tag,
is the immutable identity.

**Execution mode:** Existing-codebase `team-pr` discipline on the current
`codex/v0.45-recovery` branch: one task-owning implementer at a time, scoped
Conventional Commits, then a fresh read-only reviewer. Preserve every unrelated

## Frozen decisions and non-goals

1. An omitted repository `materialization` means the current Git-tree behavior.
   Existing v10.3.0 Git manifests remain valid and keep their current
   `sourceId` and `fileId` values.
2. A release archive requires method version `v10.3.1` or later. It does not
   bump `STRUCTURE_SCHEMA_VERSION`; this is an additive calibration-only
   contract.
3. The repository record remains tied to an immutable upstream Git commit for
   project/license context, while `materialization.assetSha256` identifies the
   actual archive bytes that contain the file.
4. Canonical release-archive source IDs use this exact form:

   ```text
   <repositoryId>@<commitSha>+asset-<assetSha256>:<normalizedPath>
   ```

5. The manifest's `normalizedPath` is relative to
   `materialization.rootPrefix`, never to the host cache. The emitted checkout
   map points at that contained root.
6. The downloader/extractor never runs inside the resolver. `scan` has no
   network access and consumes only an already verified materialization.
7. Local files, `file:` URLs, mutable/unhashed assets, unchecked redirects,
   and user-supplied extracted directories remain inadmissible.
8. This work does not decide labels, tiers, splits, duplicate safety, license
   compatibility, or release eligibility. Those remain separate provenance
   and calibration-review gates.
9. This work supports ZIP only. Tar, Git LFS, OCI, S3, model hubs, and generic
   generated-directory imports require new policy versions and separate
   threat-model reviews.

## Exact contract shape

The optional repository field is:

```ts
interface ReleaseArchiveMaterialization {
  readonly kind: 'release_archive';
  readonly assetUrl: string;             // absolute HTTPS URL
  readonly assetSha256: string;          // lowercase 64-hex
  readonly assetBytes: number;           // exact positive safe integer
  readonly archiveFormat: 'zip';
  readonly rootPrefix: string;            // safe archive-relative directory
  readonly extractionPolicy: 'safe-zip-v1';
}
```

The optional local checkout-map binding is deliberately smaller:

```ts
interface ReleaseArchiveCheckoutBinding {
  readonly kind: 'release_archive';
  readonly assetSha256: string;
  readonly extractionPolicy: 'safe-zip-v1';
}
```

Selection records copy that binding. They do not copy `assetUrl`,
`assetBytes`, or a host path; the copied canonical manifest and its hash retain
the full acquisition record.

`safe-zip-v1` is a frozen algorithm, not a prose label. It must enforce:

- at most 100,000 entries;
- at most 1 GiB total declared uncompressed bytes;
- at most 32 MiB per regular file;
- at most 200:1 declared uncompressed/compressed ratio per file and in total;
- at most 4,096 printable-ASCII bytes per canonical path, 255 bytes per
  segment, 64 segments, and 64 MiB of canonical path bytes across the archive;
- `/` separators only; no control/non-ASCII byte, NUL, backslash, leading
  slash, drive prefix, repeated slash, empty segment, `.` segment, `..`
  segment, or more than one trailing directory slash;
- no exact duplicate path and no collision after mapping ASCII `A-Z` to
  `a-z`; non-ASCII/Unicode paths and the Unicode Path extra field are rejected
  by v1 rather than decoded through runtime-dependent Unicode tables;
- compression methods 0 (stored) and 8 (deflate) only, with an explicit
  general-purpose-flag allow-list, one unique comment-free EOCD, contiguous raw
  central/local ranges, consistent raw/yauzl and central/local metadata, signed
  data descriptors only, CRC-32 verification, and bounded ZIP64-v1 preflight;
- regular files and directories only, with frozen Unix/DOS type
  classification; reject symlinks, hard-linked files, devices, FIFOs,
  sockets, volume labels, encrypted/patch/stream entries, unsafe extra fields,
  unknown host systems, duplicate local offsets, and overlapping ranges;
- a trusted, canonical, pre-existing POSIX cache root; descriptor-bound archive
  hashing before and after extraction; private random generation directories;
  `wx` file creation; explicit mode `0600` files and mode `0700` directories;
  ignored archive mtimes/permissions; and no pathname reopen of the archive;
- a deterministic receipt/inventory excluding its reserved control file,
  synced bottom-up before publication; publish only a local cache reference
  with same-filesystem hard-link-if-absent, never whole-directory `rename`;
- complete descriptor/no-follow identity and inventory re-verification before
  reuse. Native Windows reparse-point support, hostile same-euid protection,
  Unicode names, unsigned descriptors, entry-level ZIP64, and automatic orphan
  collection require later explicit policy versions.

Changing any limit or rule creates `safe-zip-v2` and a new calibration method
version. Do not silently loosen `safe-zip-v1` after observing a corpus.

### Step 3: implement fail-closed acquisition

Do not use automatic redirects or a pooled agent. Resolve every hop with the OS
resolver in IPv4-only/all-result mode, validate every answer, and preserve the
DNS hostname for Host, SNI, and certificate verification. Task 3 deliberately
rejects IPv6: an arbitrary RFC 6052 network-specific NAT64 prefix can look like
ordinary global unicast while translating to private IPv4, and the current
contract has no trusted-prefix or network-sandbox authority. Native IPv6 may be
added only by a later explicit policy/plan. Preserve the expected archive byte
count as both a 5-GiB absolute policy ceiling and an exact hard limit while
streaming. Rehash an existing cache entry from an opened nonblocking regular-
file handle on every run; compare pre-open and post-open device/inode identity.
Canonicalize a pre-existing cache directory once, validate
its complete trusted ancestor chain, and use only that canonical path
thereafter. Task 3 is deliberately POSIX-only and fails closed when effective-
uid/no-follow/nonblocking primitives are absent. A filename alone is not proof,
and an invalid pre-existing digest path fails without replacement.
Publish with same-directory `link`-if-absent rather than `rename`, because
POSIX rename may replace an existing target. On `EEXIST`, rehash the winner and
accept it only if it is valid. Remove only the task's own unique temporary file
on failure. Expose stable constant error codes/messages without interpolating
untrusted or local values.
