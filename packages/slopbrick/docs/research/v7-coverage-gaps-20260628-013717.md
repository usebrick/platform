## ⚠️ MAJOR GAP: non-JS/TS files are entirely unscanned

The current rule engine only fires on `.ts`, `.tsx`, `.js`, `.jsx`,
`.vue`, `.svelte`, `.astro`, `.html` files. **All Python, Go, Java,
Rust, Swift, Kotlin, Dart, C++, Ruby, PHP files are skipped before
any rule runs** (see `BACKEND_EXTENSIONS` early-return in
`src/engine/worker.ts:45`).

This is by design (the v0.9.2 cross-file drift detection handles
backend files via `buildPatternInventory` for service/route/ormModel
detection — but it does NOT feed into the rule engine). The cost:
the v7 calibration is **calibrating TS/JS rules only**.

## How to act on this

### For the v7 calibration RIGHT NOW

The TS/JS calibration is what we have. The partial calibration report
(v0.14.5k) is the actionable result — it tells us which of the 80
rules are useful on TS/JS, and which are INVERTED/NOISY.

### For v0.15 (post-v7)

Three directions to address the 70% gap:

1. **Python AST-based rules**: Add `tree-sitter-python` or
   `python-parser` to the parser pipeline. Then write Python rules
   for: docstring AI patterns, type-hint overuse, async/await
   boilerplate, pytest fixture over-specification, FastAPI
   template-by-template generation.

2. **Regex-only Python rules** (cheaper, no parser dep): rules that
   fire on raw text patterns. The `ai/markdown-leakage` rule
   already does this and could fire on Python. Add more like:
   - `ai/empty-docstring-spam` — triple-quote docstring with no content
   - `ai/import-block-uniformity` — all imports alphabetized
   - `ai/over-typed-hints` — every param has a type hint

3. **Drop the BACKEND_EXTENSIONS early-return entirely**: let the
   rule engine try Python/Go files. The rules that need SWC will
   silently produce 0 issues (their `analyze()` returns [] for
   non-matching AST). The rules that work on raw text will fire.
   Cost: a few extra ms per file. Benefit: 70% of the corpus
   starts being measured.

### For the self-scan (slopbrick's own repo)

The self-scan is all TS/JS, so the gap doesn't apply. The
v0.14.5j output is the correct view.

## Conclusion

The v7 calibration is meaningful for the TS/JS subset of slopbrick's
user base (most React/Next.js projects). For full coverage, v0.15
needs Python + Go support — at minimum, regex-only rules that fire
on raw text.

Until then, the partial calibration is the best we have, and the
INVERTED rules it identifies will be auto-defaultOff in the final
calibration, which is the most important signal.
