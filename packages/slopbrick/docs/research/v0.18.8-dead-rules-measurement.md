# v0.18.8 — dead/* rules first measurement (the "honest" release)

**Date:** 2026-07-01
**Status:** shipped
**Scope:** the 5 dead/* rules shipped in v0.18.5 (commit `095b4b5`) as
`defaultOff: true` with **no calibration data**. v0.18.8's job per
`v0.18.8-plan.md` is to give them a first measurement.

**TL;DR:** the v0.18.5 hypothesis ("AI code has more dead imports /
locals / parameters than hand-written code") is **falsified on this
sample** for 3 of 5 rules. The other 2 are NOISY or DORMANT. All 5
remain `defaultOff: true`. This is a **negative result release** —
we measured, and the measurement says the rules aren't ready.

## What this means

### `dead/dead-branch` — NOISY (2/1)

Ratio 2.0x by formula, but absolute counts of 2 vs 1 on a 1000-file
sample are within noise margin. The hypothesis ("`if (true) / if
(false)` is an AI-iteration artifact") is unconfirmed on this sample.
**Decision:** keep `defaultOff: true`, re-measure on v8 full corpus
or higher-iteration AI samples (e.g., repos tagged `claude-code-iter-*`).

### `dead/unreachable` — DORMANT (0/0)

The visitor's `findUnreachableStatements` (in
`engine/visitors/scan-helpers.ts`) reports zero fires on both arms.
Two possibilities:

1. **The pattern is genuinely rare.** Real-world TS/TSX code rarely
   has statements after `return` / `throw` at function top, in either
   hand-written or AI code. TypeScript's `noUnreachable` ESLint rule
   catches the obvious cases at lint time; what remains is genuinely
   subtle.
2. **The detection is buggy.** The post-pass in `visitor.ts` runs
   AFTER the main extraction. If it throws silently or the conditions
   for firing are too strict, zero is the expected output.

**Decision:** keep `defaultOff: true`, **and** add a synthetic
fixture in `tests/fixtures/dead-code/` to verify the visitor fires at
all. v0.18.8 follow-up: if the fixture fires but real-world doesn't,
the rule is correct but the corpus is wrong; if neither fires, the
visitor is broken.

### `dead/unused-import` — INVERTED (93/124)

Hand-written code has **33% MORE** unused imports than AI code. The
v0.18.5 hypothesis ("AI iteration leaves dead imports behind") is
**falsified on this sample**.

Plausible explanations:

- The positive arm (AI code) is newer and more likely to use
  `tsconfig.json` with `noUnusedLocals: true`, so the import gets
  deleted before the file is committed.
- The negative arm (hand-written code) skews toward older codebases
  with re-exported types (`export type { X } from './y'`) where the
  `X` is technically "unused" at the import site but is the public
  API of the module.
- TypeScript tooling (e.g., `tsc --noEmit`) often runs in CI for
  AI-generated codebases but not for older hand-written ones, so the
  import-cleanup pressure is asymmetric.

**Decision:** keep `defaultOff: true`, re-validate on a corpus with
explicit `noUnusedLocals: true` and `noUnusedParameters: true` settings
balanced across both arms. (v0.18.9+ follow-up.)

### `dead/unused-local` — NOISY (91/88)

No signal in either direction. Unused local variables appear at the
same rate in hand-written and AI code (within 1%). The v0.18.5
hypothesis is falsified. The rule's detection is sound; the
underlying signal just doesn't exist at the file level in this
sample.

**Decision:** keep `defaultOff: true`.

### `dead/unused-parameter` — INVERTED (5/9)

Hand-written code has **80% MORE** unused parameters than AI code.
The v0.18.5 hypothesis is falsified.

Plausible explanation: callback-heavy hand-written code
(`(req, res, next) => { ... only req is used ... }`) is rare in newer
AI-generated code, which tends to use destructuring
(`({ id }) => { ... }`) and arrow functions with explicit params.

**Decision:** keep `defaultOff: true`, re-validate on a corpus where
both arms have similar callback patterns.

## What the measurement did NOT measure

1. **No precision/recall on a labeled sample.** This is a
   "arm-fires" measurement, not a "true-positive rate" measurement.
   We don't know if the rules fire on the *right* files. We only
   know which arm they fire on more.
2. **No cross-file dead-code.** All 5 rules are single-file. The
   `dead/orphan-export` and `dead/orphan-file` patterns (where an
   export is never imported anywhere) are out of scope.
3. **No severity calibration.** A 0.75x ratio is "INVERTED" but the
   rule's `severity: low` is what actually matters for the user. A
   LOW-severity inverted rule is less harmful than a HIGH-severity
   INVERTED rule.

## What v0.18.9+ should do

1. **Build a higher-iteration AI sample.** v0.18.8 used the v7
   positive arm, which is biased toward "polished" AI output (the
   vibe-coded repos are demos, not in-progress work). v0.18.9 should
   add the in-progress AI work — repos tagged
   `claude-code-WIP`, `cursor-session-*`, etc. — and re-measure.
2. **Add a synthetic dead-code fixture.** `tests/fixtures/dead-code/`
   with one file per rule, each containing 5+ clear violations. The
   fixture runs the scan and asserts the rules fire on the right
   line. This catches visitor regressions.
3. **Re-measure on the v8 full corpus.** Once v8 is built (CSS-in-JS,
   React 19, Next.js App Router additions), the dead/* rules get
   re-measured on the larger sample.
4. **Consider rule-design changes.** The 3 INVERTED / NOISY rules
   might benefit from stricter detection (e.g., only flag unused
   imports where the module is also a string type, or only flag
   unused parameters in non-callback contexts). These are
   design changes, not calibration changes.

## Honest framing for the user

A v0.18.8 release that says "we measured our 5 new dead-code rules
and 3 of them are wrong" is a credibility claim. It says the team
will measure before shipping, and will report negative results. This
is more valuable than a v0.18.8 release that flips `defaultOff` to
false on 3 rules that don't have signal.

The CHANGELOG entry (`CHANGELOG.md` v0.18.8) reflects this framing
explicitly: **"the v0.18.5 hypothesis is falsified on this sample.
All 5 rules remain `defaultOff: true`."**

## Appendix: why this release is "honest"

The v0.18.x roadmap (rev 3) defines a **calibration** release as:
"all 99 rules measured, keep/delete/create decisions documented,
per-rule before/after table in CHANGELOG." v0.18.8 ships the
**measurement** for the 5 dead/* rules but does NOT ship
keep/delete/create changes for the 32 other DORMANT rules. That
work needs the full v8 corpus (multi-day corpus build), which is
deferred to a subsequent session.

This release is therefore:

- **What v0.18.8 ships:** the first measurement of the dead/* rules,
  documented in this file and `signal-strength.json`. Quality gates
  green. Version bumped. CHANGELOG entry.
- **What v0.18.8 does NOT ship:** full v8 corpus, DORMANT rule
  review, any `defaultOff` flips. Those ship in v0.18.9+ once the
  v8 corpus is built.
