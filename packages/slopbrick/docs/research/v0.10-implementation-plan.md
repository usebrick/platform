## Phase 1 \u2014 Tier 1A: Foundational primitives (~3 hours)

### 1.1 Halstead complexity measures (Halstead 1977)

**Files:** `src/engine/halstead.ts` (new, ~80 LOC), `src/rules/perf/halstead-anomaly.ts` (new, ~40 LOC), `tests/engine/halstead.test.ts` (new, ~50 LOC), `tests/rules/perf/halstead-anomaly.test.ts` (new, ~50 LOC).

**What it computes:** per-component `{vocabulary, length, volume, difficulty, effort, estimatedBugs}` from the AST tokens we already extract.

**Why it's foundational:** every "this component is too simple / too uniform" finding can now cite Halstead 1977 \u00a73 ("Software Science") instead of being a heuristic. Replaces threshold guessing in rules like `math-any-density` (which currently has no peer-reviewed basis for its thresholds).

**Calibration target:** USEFUL rule with P \u2265 60%, R \u2265 1%, FPR \u2264 0.1% (Halstead anomalies are rare but high-signal).

### 1.2 Cyclomatic complexity (McCabe 1976)

**Files:** extend `src/engine/halstead.ts` with `M = E - N + 2P`, add to `src/rules/perf/cyclomatic-anomaly.ts` (new, ~30 LOC), tests.

**Why it's foundational:** 10,000+ citations; the default code-complexity metric in every IDE. AI components tend to have lower M (linear "happy path" code). Cheap to compute from the control-flow graph we already build.

**Calibration target:** USEFUL rule, similar to Halstead.

### 1.3 Halstead vocabulary entropy

**Files:** inline in `src/engine/halstead.ts` (~5 LOC), add as sub-component of the volume metric.

**Formula:** `H_vocab = (n1 \u00b7 log\u2082(n1) + n2 \u00b7 log\u2082(n2)) / N`.

**Why foundational:** lower entropy = more uniform naming = AI signature. Cited from Halstead 1977 \u00a73.4.

### Commit cadence

Each engine+rule pair ships as ONE commit (`feat(v0.10): Halstead complexity + halstead-anomaly rule`). Pre-commit hook runs `npx slopbrick --staged`; calibration thresholds must pass.

## Phase 2 \u2014 Tier 1B: Code Naturalness + KL Novelty (~1 day)

### 2.1 Code Naturalness via AST n-gram entropy (Hindle 2012)

**Files:** `src/engine/naturalness.ts` (new, ~120 LOC including fitting the n-gram model against the v4 corpus baseline), `src/rules/visual/naturalness-anomaly.ts` (new, ~40 LOC), tests.

**What it computes:** per-component cross-entropy of the AST-tok sequence against an n-gram model fit on the v4 corpus baseline. High cross-entropy = component deviates from corpus distribution = likely AI.

**Why it's foundational:** Hindle 2012 is the most-cited paper in code intelligence (4,000+ citations). The 2024 TSE extension "Dependency-Aware Code Naturalness" makes it AST-aware, which matches our architecture.

**Calibration target:** USEFUL rule with P \u2265 70% (Hindle 2012 reports precision ~75% on their corpora).

### 2.2 KL divergence pattern novelty (Kullback & Leibler 1951)

**Files:** `src/engine/kl-novelty.ts` (new, ~60 LOC), integrate as new axis in `src/engine/architecture-score.ts`, tests.

**What it computes:** for each pattern category (state lib, form lib, modal pattern), `KL(P_project \u2225 P_corpus)`. High KL = project uses patterns uncommon in human corpus = statistical surprise.

**Why foundational:** Kullback-Leibler divergence is foundational (15,000+ citations). Gives a principled threshold for "this project uses patterns that are statistically surprising" \u2014 the heart of cross-file drift.

**Calibration target:** the `Architecture Consistency` score's behavior should correlate with the v4 calibration's drift detection rate (currently high precision on cross-file drift but no formal basis).

## Phase 3 \u2014 Tier 1C: MDL Composite \u2014 the principled score (~1 day)

**Files:** `src/engine/mdl.ts` (new, ~200 LOC), update `src/engine/repository-health.ts` to use `Coherence_MDL` as the headline axis, tests.

**What it computes:** `Coherence_MDL(file) = log P(rules_fired | m_ai) - log P(rules_fired | m_human)` where `m_ai` and `m_human` are two competing models of code provenance trained on the v4 corpus.

**Why it's the credibility milestone's centerpiece:** the current `Repository Coherence Score` is a heuristic weighted average (0.50 \u00b7 arch + 0.30 \u00b7 fragmentation + 0.10 \u00b7 constitution + 0.10 \u00b7 ai-debt). The weights were chosen empirically. MDL gives a principled derivation: the score is the log-likelihood ratio of "all this evidence under M_ai" vs "under M_human". Engineers can argue the model (which rules belong to M_ai vs M_human), not the weights.

**Calibration target:** MDL-derived score should correlate with existing P/R/FPR \u2265 0.7. The m_ai / m_human models are calibrated against the v4 corpus.

**Citation chain:** Rissanen 1978 (MDL principle), Grunwald 2007 (textbook), Grunwald 2019 review.

### Commit

Single large commit: `feat(v0.10): MDL composite score replaces heuristic weighted average`. Includes model definition (which rules belong to m_ai vs m_human), fit procedure, calibration report.

## Phase 6 \u2014 Tier 2 graph-theoretic (post-v0.10)

NOT in v0.10 \u2014 these land AFTER v0.10 is stable, possibly as v0.11 or v0.12 patches. They're listed here so the engineering vision is complete; committing to them in v0.10 would push their stability testing into the v1.0 freeze window, which is exactly backwards.

### 6.1 Louvain community detection on import graph (Blondel 2008)

**Files:** `src/engine/louvain.ts` (~50 LOC), integrate into `Architecture Consistency Score` as new "module structure" axis.

**What it computes:** modularity-maximizing partition of the import graph. Files that are outliers in their community = drift signal.

### 6.2 Spectral graph theory \u2014 Fiedler value

**Files:** inline in `src/engine/louvain.ts` or new `src/engine/spectral.ts` (~40 LOC).

**What it computes:** second-smallest eigenvalue of the import-graph Laplacian. Low value = fragmented modules = drift.

### 6.3 Bayesian Online Changepoint Detection (Adams & MacKay 2007)

**Files:** `src/engine/changepoint.ts` (~150 LOC).

**What it computes:** regime changes in rule-firing rate over lines of a file or commits over time. Detects "this PR was authored under a different regime than the rest of the file" \u2014 likely AI-assistance mid-edit.

## Phase 7 \u2014 Repository Memory Platform (far horizon, prerequisite for --diff and BRICK)

Strategic feature. Not in v0.10 but prerequisite for the BRICK platform vision.

### 7.1 `.slop-audit/inventory.json` + `.slop-audit/constitution.json`

**Files:** `src/engine/memory.ts` (new, ~100 LOC), scan-pipeline integration, tests.

**What it does:** every `slopbrick scan` persists its extracted patterns (modal, button, state, api, data-fetching) and declared constitution to disk. Subsequent MCP tool calls read from disk instead of re-scanning.

**Why it matters:** `slop_suggest` currently re-scans on every call. With persisted memory, MCP calls are O(read JSON) instead of O(re-parse AST). This is a 100\u20131000\u00d7 latency improvement for the agent integration.

### 7.2 `.slop-audit/memory.md`

**Files:** `src/engine/memory-md.ts` (new, ~60 LOC).

**What it does:** human + agent-readable summary of the patterns. Used by `slop_suggest` to return canonical patterns to the agent in markdown form.

### 7.3 Incremental refresh (mtime check)

**Files:** extend `src/engine/memory.ts`.

**What it does:** before re-scanning a file, check mtime against the persisted inventory hash. Skip re-extraction if unchanged. Plugs into the existing `--incremental` flag.

## Phase 8 \u2014 `--diff [ref]` flag (after Phase 7)

**Files:** extend `src/cli/program.ts` + `src/cli/scan.ts`.

**What it does:** compare current working tree against a git ref. Returns the delta in pattern inventory, constitution drift count, and per-rule PR slop score for the diff. Competitor response to VibeDrift's `--diff` flag.

---

## Phase 9 \u2014 `find_similar_function` (after Phase 7)

**Files:** `src/engine/find-similar.ts` (new, ~150 LOC), MCP tool exposure.

**What it does:** given a function/hook signature, find the most similar existing implementations across the codebase. Uses AST fingerprints (no LLM, no embeddings \u2014 just hash-based tree similarity per Chilowicz 2009 syntax-tree fingerprinting).

**Why it matters:** enables GIR (Give-Implementation-Reference) pattern in `slop_suggest`. Foundation for StackPick.

---

## Phase 10 \u2014 BRICK Platform (after Phases 7\u20139)

**Files:** separate package / monorepo structure.

**What it ships:**
- **StackPick:** the installable package; reads `.slop-audit/` memory + adds curated stack detection on top.
- **GIR (Give-Implementation-Reference):** AI agent integration; calls `find_similar_function` before writing new code; uses `slop_suggest` for canonical patterns.
- **Enforcement:** CI gate that fails PRs introducing new patterns when the existing inventory covers them.

## What this plan deliberately leaves out

1. **v1.0 framing** \u2014 far horizon, per user direction. Don't promise v1.0 in commit messages or release notes.
2. **Deep learning stylometry** (Bisztray 2025) \u2014 too compute-heavy per the math-foundations doc.

4. **More heuristic rules** \u2014 we have 18 USEFUL rules with calibration; adding more without P/R/FPR data is moving backwards.
5. **Refactors that don't move the credibility needle** \u2014 most engine cleanup is already done; what's left is calibrated, peer-reviewed, or skipped.

## How to use this plan

- Pick one phase at a time. Don't try to parallelize via subagents \u2014 the academic methods compose with each other and the order matters.
- Each phase lands as one or more commits. Each commit message references which rule now cites which paper (for Phase 1\u20133).
- Re-baseline `pnpm vitest run` after each phase. Per-rule P/R/FPR shifts as new rules land; the v0.10 readiness checklist explicitly tracks this.
- After Phase 5 ships v0.10, the user picks Phase 6 vs Phase 7. Both have strategic value; the user's call which is next.
