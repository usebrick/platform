## Why this is needed

v0.9.1's 3 service-proliferation rules (`multiple-state-systems`, `multiple-modal-systems`, `multiple-api-clients`) detected **intra-file** mixing ‚Äî "this file uses 2+ libraries." That's not the lens.

The lens is **cross-file** drift: "repo already had UserService, AI created UserManager ‚Äî that's drift." Detecting it requires:

1. **Extract patterns** from every file (Python classes, Go structs, etc.) ‚úì DONE
2. **Aggregate into an inventory** (already exists as `buildPatternInventory` in `src/mcp/patterns.ts`) ‚úì DONE
3. **Cluster by similarity** ‚Äî `UserService` and `UserManager` are two patterns with the same `User` stem but different suffixes. **TODO phase 3**.
4. **Detect drift** ‚Äî when a new pattern emerges alongside an established one, flag it. **TODO phase 4**.

## The lens in one line

> If the flagship metric is Repository Coherence, most rules are **language-agnostic**. You don't care whether it's `class UserService` or `type UserService struct`. You care that the repo already had `UserService` and AI created `UserManager`.

## Scope (3 weeks, split across sessions)

### Phase 1 ‚Äî Backend file discovery + skeleton extraction (this or next session, ~3 hours)

The engine currently skips non-frontend files. Add Python + Go + Ruby + Java files to the discovery loop, but with **minimal extraction**:

- Python: regex-based extract of `class \w+(\(.*?\))?:`, top-level `def \w+\(`, imports
- Go: regex-based extract of `type \w+ struct`, `func (\w+) \w+\(`, imports
- Ruby: regex-based extract of `class \w+`, `def \w+`, requires
- Java: regex-based extract of `class \w+`, `public \w+ \w+\(`

No full AST parsing yet ‚Äî regex is enough to seed the inventory with named entities. Real AST parsing comes in Phase 4 if needed.

### Phase 2 ‚Äî Extend `buildPatternInventory` (~2 hours)

Add new pattern categories to `PatternInventory.patterns`:

```ts
patterns: {
  modal: PatternMatch[];        // existing (frontend)
  button: PatternMatch[];       // existing (frontend)
  api: PatternMatch[];          // existing (frontend)
  state: PatternMatch[];        // existing (frontend)
  dataFetching: PatternMatch[]; // existing (frontend)
  // NEW for v0.9.2:
  service: PatternMatch[];      // backend service classes / structs
  route: PatternMatch[];        // HTTP route handlers (any framework)
  ormModel: PatternMatch[];     // SQLAlchemy / Django / gorm / sqlx models
};
```

Each category is populated by language-specific extractors feeding the same `PatternMatch` shape. The architecture-consistency score (which already consumes this inventory) now sees backend patterns.

### Phase 3 ‚Äî Name-similarity clustering (~3 hours)

For the cross-file drift detection, we need to group similar-named entities:

```
UserService, UserManager, UserHandler, UserRepository
  ‚Üì strip common suffixes
User
  ‚Üì 4 distinct forms of "User" = drift
```

Algorithm:
1. For each pattern category, collect all entity names (e.g., all `service` names)
2. Strip common suffixes: `Service`, `Manager`, `Handler`, `Repository`, `Controller`, `Helper`, `Util`, `Factory`, `Provider`, `Store`, `Repo`, `API`, `Client`
3. Group by stem
4. A stem with 2+ distinct suffix patterns = drift

Output: a new `coherenceDriftSignals` field on the report listing the drifted stems and their variants.

### Phase 4 ‚Äî Cross-file drift rule (~3 hours)

A single new rule (or signal ‚Äî could be embedded in the architecture-consistency engine):

> **`arch/cross-file-pattern-drift`** ‚Äî fires when the inventory shows 2+ distinct patterns for the same conceptual entity (UserService vs UserManager for "User"). One issue per drifted stem, with file list and pattern variants.

### Phase 5 ‚Äî Go visitor proper (regex ‚Üí optional AST) (~1 week)

Go has stricter type semantics than Python. The Go visitor should extract:
- `type Foo struct { ... }` ‚Äî service structs
- `func (f *Foo) Method()` ‚Äî methods
- `type Foo interface { ... }` ‚Äî interfaces
- `http.HandleFunc("/path", handler)` ‚Äî routes
- `db.Query(...)`, `db.Exec(...)` ‚Äî DB calls
- import paths

Consider `tree-sitter-go` for proper AST. But regex-first is fine for v0.9.2.

### Phase 6 ‚Äî Tests + calibration (~1 week)

- 10 hand-picked Python repos for the labeled dataset (Python is the bigger corpus; we already have it at `/Users/cheng/corpus-expansion/`)
- 10 hand-picked Go repos
- Compute empirical P/R/FPR on the new rules
- Re-classify the new rules into Core Coherence / Supporting / Independent Domain
