# v0.18.9 — Real v8 corpus: refresh + tree-sitter for Rust

**Status:** in flight
**Scope:** build a real v8 corpus (replacing the v0.18.8 "v8a" 1000-file
subset), refresh both arms with proper age separation, add tree-sitter
support for Rust as the first new backend language, re-calibrate all
99 rules + the 5 dead/* rules on the new corpus.

**Out of scope (deferred to v0.19+):** tree-sitter for Java, Ruby,
Kotlin, Swift, Dart, C++, PHP. The regex-stub visitors stay in
place but don't produce rule signal.

## Why this is a separate release (not a v0.18.8 redo)

v0.18.8 shipped with a 1000-file v7-subset. The user (correctly)
flagged that this is not a real v8 calibration. v0.18.9 is the
**real** v8: fresh corpus, fresh scan, fresh verdicts.

## User's three decisions (from `AskUserQuestion`)

1. **Scope:** TS/TSX + Python + Go corpus refresh + tree-sitter
   for at least one backend language (Rust picked — see below).
2. **Negative arm:** both strategies — `git checkout` to old
   commits of active repos + abandoned repos that stopped
   updating pre-2023.
3. **Positive arm:** both strategies — claude-code/cursor/copilot
   workspace exports + v0/lovable/bolt.new outputs.

## Why Rust first (not Java, Ruby, etc.)

Rust is the first tree-sitter target for v0.18.9 because:

1. **Mature grammar:** `tree-sitter-rust` is one of the best-maintained
   grammars (rust-lang/tree-sitter-rust, used by nvim, helix, Zed).
2. **AI tooling support:** claude-code, cursor, bolt.new all
   generate Rust well; the positive arm is easy to source.
3. **Dead-code rules are especially relevant:** Rust's compiler
   already warns on local dead code, but cross-file dead code
   (orphaned functions, unused public exports) is what slopbrick's
   dead/* rules can find. Real measurement on real Rust code.
4. **Smaller corpus target:** Rust repos are typically smaller than
   Java/Spring; the calibration can run on 2-3k Rust files.
5. **Existing visitor stub:** `src/engine/visitors/rust.ts` already
   has the regex-based service/route extraction. The new tree-sitter
   integration extends this rather than replacing it.

Java, Ruby, Kotlin, Swift, Dart, C++, PHP get tree-sitter in v0.19+.
The order is: Java (largest enterprise codebase), Ruby (Rails
ecosystem), Kotlin/Swift (mobile, matched to existing Flutter/RN
negative arm), then the rest.

## Tree-sitter integration (Rust first)

### Dependencies

- `tree-sitter` (the core library, npm package `tree-sitter` ~v0.20)
- `tree-sitter-rust` (Rust grammar)
- Add both to `packages/slopbrick/package.json` dependencies

### Files to add / modify

- `packages/slopbrick/src/engine/parser-rust.ts` — new file, wraps
  tree-sitter-rust. Produces a minimal AST shape compatible with
  the existing visitor.
- `packages/slopbrick/src/engine/visitors/rust.ts` — extend the
  existing regex visitor to use tree-sitter when available, fall
  back to regex if tree-sitter fails.
- `packages/slopbrick/src/types/scan.ts` — extend `ScanFactsV2` to
  include Rust file structure (functions, structs, traits, impls).
- `packages/slopbrick/src/rules/dead/*.ts` — add a `rust` case to
  the file-extension check so the dead/* rules apply to Rust too.

### New Rust rules (target: 3-5)

- `rust/unused-pub-fn` — public function with no callers in the corpus
- `rust/unwrap-in-production` — `unwrap()` outside of `test` cfg
- `rust/todo-macro` — `todo!()` / `unimplemented!()` in production
- `rust/stringly-typed` — `&str` or `String` used where a typed enum exists

These are scoped to Rust's specific failure modes, not the same
generic "dead code" detection. They calibrate the tree-sitter
integration rather than duplicating the v0.18.8 dead/* rules.

## Phases

### Phase 1 — Plan (this doc + repo list)

This document. The repo list above is the v8 target.

### Phase 2 — Pull negative repos (one session)

For each repo, `git clone --depth=1` + `git fetch --unshallow` +
`git checkout <old-sha>`. Or for shallow-only repos, `git clone
--branch <tag> --depth=1` (but tags don't always work, so
unshallow + checkout is more reliable).

Run the existing `build-filelists-v2.sh` (extended for the new
repos) to regenerate filelists.

### Phase 3 — Pull positive repos (one session)

For each AI tool source, clone the public exports. Many of these
are monorepos — use `find` to extract only the relevant files.

### Phase 4 — tree-sitter integration for Rust (1-2 sessions)

Add `tree-sitter` and `tree-sitter-rust` to dependencies. Wire
into the parser. Add the 3-5 Rust-specific rules. Test on a
synthetic Rust fixture.

### Phase 5 — Full v8 scan + calibration (one session, 4-6 hours)

Run the existing `scan-corpus-robust-v2.ts` on the new v8 corpus.
The existing `compute-v7-calibration.py` reads the scan output and
produces per-rule P/R/FPR.

### Phase 6 — Apply verdicts, ship v0.18.9 (one session)

- Update `signal-strength.json` for all 99 + 5 dead/* + 3-5 new
  Rust rules.
- `defaultOff` flips based on verdicts:
  - USEFUL/OK: keep `defaultOff: false`
  - NOISY/HYGIENE/INVERTED: keep `defaultOff: true`
  - DORMANT: review for deletion (Phase E from the v0.18.8 plan)
- CHANGELOG with the per-rule "was X, now Y" table.
- Version bump 0.18.8 → 0.18.9.
- Tag + release + publish workflow.

## Open questions

1. **Should the Rust rules use the v0.18.5 dead/* framework, or be
   new Rust-specific rules?** This plan proposes new Rust-specific
   rules (`rust/unwrap-in-production` etc.) because the dead/* rules
   are file-scoped and Rust's dead-code is mostly compiler-detected.
   The dead/* rules become useful for cross-file dead Rust code
   (orphaned pub fns) in a later release.
2. **What corpus cap?** 5-8k per arm is the target. The scan can
   handle 50k+ files per arm but the v7 184k+239k proved we don't
   need that many. 10-15k total is plenty for stable per-rule
   calibration.
3. **What about the abandoned-repo strategy?** The plan mentions it
   but doesn't list specific repos. Examples: `jquery/jquery`
   (mature, dormant), older Vue 2 ecosystem repos, `molefrog/wouter`
   (simple hand-written router, 2020-2022), `kettanaito/nock`
   (HTTP mocking, 2018-2020). Will refine in Phase 2.
