## Phases

### Phase A — Plan + setup (this doc)

This document. It is the entry point for v0.18.8; the roadmap
(`v0.18-roadmap.md`) has the high-level goals.

### Phase B — Build the dead/* calibration sample

Source ~2,000 files: ~1,000 AI-coded repos known to produce dead code
(empty try/catch, unused imports, defensive branches) and ~1,000
hand-written repos where dead code is rare (mature frameworks, libraries).

The 1:1 ratio matters because dead-code rules have low absolute recall
— we need enough negatives that any DORMANT verdict is meaningful, not
a sample-size artifact.

**Output:** `packages/slopbrick/data/v8-dead-corpus.ts` (a static
sample list, like `tests/fixtures/frameworks/`), plus the filelists
entry in `/Users/cheng/corpus-expansion/v8/filelists/`.

### Phase C — Scan + calibrate

Run the existing scan script (or a focused version) on the v8 sample
for **only the dead/* rules** (5 rules × 2,000 files = 10,000
rule-applications; takes minutes, not hours).

Output: per-rule TP/FP/P/FPR for the 5 dead/* rules. This is the
first-ever measurement.

**Output:** `packages/slopbrick/src/rules/signal-strength.json` (add
5 entries), `docs/research/v0.18.8-dead-rules-measurement.md` (the
report).

### Phase D — Decide for the 5 dead/* rules

Based on Phase C:

- **All 5 USEFUL / OK:** flip `defaultOff: false` on the survivors.
  This is the **first time dead/* rules become user-visible by default**.
- **All 5 DORMANT / NOISY / INVERTED:** keep `defaultOff: true`, but
  the calibration entry in `signal-strength.json` is the proof — future
  versions can re-check.
- **Mixed:** flip the useful ones, document the reasoning for the
  dormant ones.

### Phase E — Decide for the 32 DORMANT non-dead rules

The 32 DORMANT rules (other than dead/*) are the v0.18.8 "kill or
keep" decision. Each gets one of:

- **Keep — corpus gap:** rule is sound but the v7 corpus doesn't have
  the pattern. Add to the v8 corpus and re-run.
- **Keep — false negative on negative arm:** rule fires on a
  hand-written pattern that doesn't appear in v7's positive arm.
  Document the corpus pattern.
- **Delete:** no signal in v7 or v8, no theoretical basis. Move to
  `src/rules/_archive/` (or delete if no dependents).
- **Repurpose:** the rule's underlying check is useful in a composite
  rule (e.g. two DORMANT rules combine into one USEFUL composite).

This is a per-rule review, not a one-shot operation. The output is
a table in the CHANGELOG.

### Phase F — Ship

- Bump version `0.18.7` → `0.18.8`
- CHANGELOG entry: the per-rule "was X, now Y" table from Phase D + E
- All quality gates green
- Tag + publish

## Review gates (from `v0.18-roadmap.md`)

Every PR in v0.18.8 must pass the blocking gates:

- **G1 (Reproduction):** the dead/* "first measurement" claim must
  cite the actual TP/FP numbers from the v8 scan output. Not an
  estimate.
- **G2 (Target verification):** the cited rule files
  (`src/rules/dead/*.ts`) and the calibration script
  (`scripts/compute-v7-calibration.py`) must be the real ones, not
  memory.
- **G3 (Fix-locus):** the dead/* `defaultOff` flip is a registry
  change, not an engine change. The fix is in `src/rules/dead/*` and
  `src/rules/registry.ts`, not in `src/engine/`.
- **G4 (Test):** a test asserts the post-flip behavior on the same
  fixtures that produced the pre-flip measurement.
- **G5 (No regression):** full `vitest run` plus `bench:scan` (the
  4-score gate).
- **G6 (Schema coherence):** the new `signal-strength.json` entries
  match the existing format (verified against the v7 entries).
- **G7 (Doc/code agreement):** the CHANGELOG, the docs/research
  report, and the rule's `description` field agree.
- **G8 (Reproduction in entry):** the CHANGELOG entry includes the
  per-rule before/after numbers.
