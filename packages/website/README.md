# usebrick.dev — marketing site

The brick-themed landing page for the slopbrick monorepo. Astro with native
IntersectionObserver/Web Animations APIs and a CSS brick surface.

## Dev

```bash
# From the monorepo root (the dev script regenerates release facts first):
pnpm --filter @usebrick/website dev
# → http://localhost:4321
```

## Build

```bash
pnpm --filter @usebrick/website build
# → packages/website/dist/
```

The `prebuild` script (`scripts/prebuild.ts`) reads the sibling package and
registry metadata and writes both `src/data/version.json` and
`src/data/product-facts.json`. The `dev` script runs this generator before
Astro starts, so a clean checkout does not depend on a previously generated
untracked data file.

## Architecture

- **Astro 7** with `output: 'static'` for SSG
- **Astro Islands** for the few interactive pieces:
  - `Hero.astro` — static brick surface and install prompt
  - `break-on-hover.ts` — click-to-break feedback on tool cards
- **Native browser APIs** for smooth-scroll behavior, reveals, counters, and
  click feedback; no Lenis or GSAP runtime dependency is required.
- **Tabler Icons** for the small icon set (file, copy, etc.)

## Theme

The brick theme is defined by CSS custom properties in
`src/styles/global.css`. The brand is **terracotta on warm dark
mortar** — `#dc4a26` accent on `#1a0e08` background. All semantic
state colors (success, warning, danger) are warm tones that fit the
brick palette rather than the default red/green/blue.

## File structure

```
src/
├── pages/
│   ├── index.astro          # landing page
│   ├── docs/index.astro     # docs route and canonical links
│   ├── changelog/index.astro# changelog route and canonical link
│   └── 404.astro            # real static unknown-route response
├── layouts/
│   └── Base.astro           # shared <head>, font loading, and global scripts
├── components/              # one .astro per section
├── scripts/                 # TS modules (native interactions, reveals, counters, copy)
├── styles/                  # global.css (tokens), theme.css, components.css
└── data/                    # version.json + product-facts.json (generated by prebuild)
```

## Deploy

The site is prepared for **Cloudflare Pages** (project `platform`, custom
domain `usebrick.dev`). This checkout contains no proof of a current
production deployment; a release owner must run and verify the documented
Cloudflare deployment flow before calling the site live. `packages/website/dist/`
can be uploaded with `wrangler pages deploy` after the local build and preview
gates pass (see [`docs/operations/cloudflare-pages-setup.md`](../../docs/operations/cloudflare-pages-setup.md)).

GitHub Actions (`.github/workflows/ci.yml` + `publish.yml`) handle the
**other** packages:
- `ci.yml` — typecheck + test on every PR / push to main
- `publish.yml` — `release: types: [published]` → build → npm publish
  slopbrick (two human gates: the environment approval, and the
  release itself).

The candidate version is generated from `../slopbrick/package.json` at build
time. Published-version facts are kept separately in
`src/data/product-facts.json`; do not label a candidate as shipped until npm
and Cloudflare evidence exists.
