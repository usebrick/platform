## 0.41.0 — Temporal drift + compositeScore surfaces (Sprint 2 wrap)

### Architectural foundations (Sprint 2 / §2)

- **`exitOverride` dispatcher** in `src/cli/commands/_shared.ts`. Single canonical Commander.js pattern: `setExitOverride` + `dispatch` + `withExitCode`. Replaces 59 of 75 inline `process.exit(N)` calls in command action callbacks. The pure-function entry points (`runDrift`, `runPrScan`, etc.) are now reachable from integration tests without spawning a subprocess.
- **`runSuggest` + `runSuggestWithStructure` consolidation** (`F4`) — one consolidated function with `includeStructure: boolean` opt-in. Backward-compat shim re-exports the legacy name.
- **Engine-purity comment clarified** in `src/engine/worker.ts:13` — explicitly attributes purity to the `packages/engine/` package, so the slopbrick `worker` (an explicit I/O boundary) is correctly identified as the boundary that threads env vars into `ParserCacheConfig`.
- **`scans.jsonl` inventory summary** — each telemetry payload now carries a compact per-category pattern summary (`scannedFiles` + `patternCounts` + `patternNames`, capped at 50 per category). Self-sufficient for `slopbrick drift --temporal-since <date>` queries without re-scanning.

### Temporal drift — `slopbrick drift --temporal-since <date>` (§2a)

- New flag `--temporal-since <iso-date|'baseline'>` on the `drift` subcommand. Computes the pattern set-difference between the most-recent scan and a baseline scan (oldest scan at-or-after `<iso-date>`, or the literal string `'baseline'` for the oldest scan in `scans.jsonl`).
- New pure functions in `src/cli/drift.ts`:
  - `runDriftOverTime(cwd, config, { since })` — pure, no I/O outside reading `scans.jsonl`.
  - `formatDriftOverTime(result)` — pretty-printed report.
  - `driftOverTimeExitCode(result)` — returns 0 when nothing introduced is undeclared, 1 otherwise (CI-friendly).
  - `flattenPatternNames(...)` and `collectDeclaredNames(...)` — exported for tests.
- Cross-checks introduced patterns against `config.constitution` and surfaces the undeclared subset as `introducedUndeclared`. Conservative case-insensitive name match across `stateManagement`, `dataFetching`, `uiLibrary`, `forms`, `styling`, `routing`, `custom.*`, and `forbidden`.
- Naming note: the §2a.5 plan called this flag `--since <date>`, but `--since` and `--baseline` were already taken by global scan flags in `program.ts`. Renamed to `--temporal-since` to keep both surfaces disjoint without changing the argument type.

### compositeScore surfaces — pretty, sarif, mcp (§2b)

- The project-level Bayesian aggregate (`compositeScore`, computed since v0.18.2 and previously only visible as a post-scan stderr log line) is now reachable through three named user surfaces:
  - **Pretty reporter** (`formatCompositeScore` in `src/report/pretty.ts`) — appends a `composite=<tier>@<mean>` line below the headline scores, mirroring the existing `persistRun.ts:230` log shape. Pluralizes `1 file` vs `2 files`.
  - **SARIF tool driver** (`src/report/sarif.ts`) — surfaces `properties.compositeScore` on the SARIF tool driver per SARIF 2.1.0 §3.18. SARIF consumers (GitHub code scanning, IDE security panels) can display the project-level probability alongside per-result findings.
  - **MCP `slop_suggest`** (`src/mcp/tools.ts`) — adds `compositeScore` to the `runSuggest` payload via `loadHealth(cwd)`. Graceful fallback when `health.json` is missing or pre-v0.18.2 (no composite field).

All three surfaces gate on `report.compositeScore` being defined. Pre-v0.18.2 reports keep their original output verbatim (no `composite=` line, no SARIF `properties` key, no `compositeScore` field).

### Scope divergence from plan

The MCP `compositeScore` payload omits `topContributors` (rule ranking by log-likelihood contribution). The persisted `HealthFile` shape doesn't carry it. Adding `topContributors` requires bumping `@usebrick/core`'s `STRUCTURE_SCHEMA_VERSION`, which AGENTS.md gates on a breaking change. The slot lands in v0.42.0 with the empirical-composites engine (§3b), which produces ranking data natively via the `TriggeredRule` log-likelihoods.
