## 0.40.0 — Self-calibration loop (relaxation half)

### Self-calibration loop

- **`severityRelax()` helper** in `src/engine/flywheel.ts`. Floor at `'off'`, sibling to the existing `severityBump()`. Public API.
- **Auto-relaxation emission** — `computeFlywheelOutput` now emits `autoRelaxed: AutoRelaxedRule[]` alongside `autoTuned`. A rule that has stayed in `topOffenseIds` across 5+ consecutive scans gets relaxed one step (`high → medium → low → off`).
- **Read-side wiring** in `src/cli/scan.ts:299` applies the relaxed severity to `config.rules[id]` on every subsequent scan, parallel to the existing bump loop. The same `defaultOff` guard (v0.14.5g) protects both halves — calibration-failed rules are never relaxed.
- **Persistence** in `src/cli/report/persistRun.ts` writes both `autoTuned` and `autoRelaxed` to `.slopbrick/flywheel/auto-tuned.json` together.

### Schema

- `FLYWHEEL_VERSION` `'2' → '3'`. `migrateFlywheelState` defaults the new field to `[]` for v0.39.x state files; v3 files persist both lists.
- New `AutoRelaxedRule` interface carries `severity` (which is `Severity | 'off'`), `previousSeverity`, `reason`, `defaultPrior`, `relaxedAt`.

### Notes

- Bumping and relaxing are two **independent** signals on top of the same data. A rule that triggers both ends up in both lists in the persisted state; the read-side applies both in sequence (relaxation wins last because it's processed second). The `Reason` field on each entry explains which observation triggered it.
- The relaxation threshold (5 consecutive unscoped scans) is the new `IGNORE_THRESHOLD` constant in `src/engine/flywheel.ts`, exposed alongside the existing `CONSECUTIVE_THRESHOLD = 3` (bump).
